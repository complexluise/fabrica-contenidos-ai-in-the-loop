## D-031 — Studio local: app web local sobre el pipeline (FastAPI + Svelte, sin auth)
**Estado:** Vigente · **Fecha:** 2026-06-05

- **Contexto:** El CLI sirve a quien sabe terminal; pero el **equipo** (narradora, diseñadora,
  ingeniero de IA, editora — ver `docs/flujo-de-trabajo.md`) necesita una **UI** para operar los
  checkpoints de [D-021]. Se decidió hacerlo **local** (corre en la máquina del usuario): eso
  **elimina** auth, usuarios, base de datos y nube — *keep it simple*. El estado sigue en disco.

- **Decisión:** Una **app local**: servidor **FastAPI** + UI **Svelte**; el navegador en `localhost`
  es la interfaz. `pipeline studio` levanta el server y abre el browser.
  - **La API es una cáscara fina sobre el motor** (`studio`/`runner`/`export`): el CLI y la API
    llaman a **las mismas funciones** (refuerza [D-023] — el contrato es la lógica, no el shell).
  - **Sin auth / usuarios / Postgres / object storage:** single-user local; el estado vive en
    `projects/`, `runs/`, `cache/` (disco), como hoy.
  - **Jobs async en proceso** (el motor ya es asyncio): cada `cast`/`keyframes`/`render` es una
    **tarea con id**; el progreso se transmite **en vivo por SSE**, reutilizando el logging de
    [D-027]. **Paralelo entre jobs** con un semáforo de concurrencia (**Fase 2**). **Planos
    concurrentes dentro del render: diferido** (Fase 3).
  - **API keys desde la UI** (pantalla de Ajustes): se escriben a un config local; se invalida el
    `lru_cache` de `settings` al guardar.
  - **Las pantallas = los checkpoints ([D-021]):** proyecto/guion · **picker** (reemplaza la hoja de
    contactos) · render (progreso vivo) · export · ajustes.

- **Consecuencias:**
  - Es un **track nuevo** (la app), con su **propio ROADMAP** (`app/ROADMAP.md`). El pipeline
    (motor) **no cambia su contrato**; gana endpoints. Backend en `src/pipeline/server/`, UI en `app/`.
  - **Reutiliza todo:** `studio`/`runner`/`export` + el logging + el caché + el bundle. La hoja de
    contactos HTML ([D-022]) era el **prototipo** del picker.
  - **AI-in-the-Loop hecho producto:** la UI **es** el loop — donde la persona decide y firma.

- **Quitado (navaja de Ockham, por ser local):** auth, multi-tenancy, Postgres, S3, deploy/ops.

- **Cambios:** Nuevo `src/pipeline/server/` (FastAPI + job manager) + `app/` (Svelte) + subcomando
  `pipeline studio`. ROADMAP propio en `app/ROADMAP.md`.

## D-032 — Estado del proyecto: DERIVADO del disco, nunca almacenado
**Estado:** Vigente · **Fecha:** 2026-06-06

- **Contexto:** La UI necesita saber "dónde estoy en el bucle" (faltan claves · casting · encuadres ·
  render · paquete) para guiar a quien llega de cero (Fase 1.5). La tentación es guardar un campo
  `status` y transicionarlo como una máquina de estados clásica. Pero el proyecto ya define su verdad
  en **artefactos en disco** ([D-013]…[D-015]: caché content-addressed, runs inmutables, resumible).
  Un `status` almacenado **mentiría**: bumpeás un `seed` o borrás un keyframe y el campo queda
  desincronizado con lo que realmente hay.

- **Decisión:** El avance del proyecto es un **estado derivado**, no almacenado. Una sola función
  pura — `state.compute_stage` (la máquina) + `state.derive_state` (la lectura de disco) — calcula el
  `Stage` del bucle a partir de los artefactos (`casting.yaml`, `selections.yaml`, `candidates.yaml`,
  `runs/`, `export/`) más el prerequisito de claves. **Fuente única de verdad**: el endpoint
  `GET /api/projects/{slug}/status` la expone y el front la **consume** (no recalcula).
  - **Los guards viven en el orden de `compute_stage`:** no se llega a `RENDER` sin `ENCUADRES`, ni a
    `PAQUETE` sin `RENDER`. Sin escenas/personajes, esos pasos se saltan solos.
  - **Distinción clave:** los **jobs** ([D-031]) sí son una FSM legítima (ciclo de vida en memoria
    `QUEUED→RUNNING→DONE/FAILED`, con efectos). El **avance del proyecto** no: es idempotente y
    re-derivable, así que **no** se le pone una FSM persistida.

- **Consecuencias:**
  - El estado nunca miente y es **resumible** por construcción (coherente con la filosofía: re-correr
    es gratis, el caché manda). No hay migraciones de estado ni reconciliación.
  - **Cero duplicación:** la lógica de completitud estaba copiada en 3 lugares (endpoint Python +
    `nextStep` + `stageState` en JS). Ahora la **decide el motor**; el front solo mapea `stage` a
    copy (`nextStep`) y a "hecho/pendiente" por orden (`stepDone`).
  - Lógica pura del core -> **testeada** (`tests/test_state.py`: transiciones + guards).

- **Cambios:** Nuevo `src/pipeline/state.py` (`Stage`, `compute_stage`, `derive_state`) + tests.
  `/status` reescrito como cáscara fina sobre `derive_state`. Front (`studio.svelte.js`,
  `App.svelte`, `Inicio.svelte`, `Produccion.svelte`) consume `stage`/`stepDone` en vez de recalcular.

## D-033 — Entrada desde la app: importar texto -> storyboard editable (Checkpoint #1 hecho UI)
**Estado:** Vigente · **Fecha:** 2026-06-06

- **Contexto:** Arrancar un proyecto exigía escribir `project.yaml` a mano — barrera para el equipo
  no-técnico ([D-031]). El Checkpoint #1/#2 de [D-021] (la IA descompone, la persona firma) no
  estaba en la interfaz. Faltaba además el **escritor** del spec: el motor sabía leer `project.yaml`
  pero no escribirlo.

- **Decisión:** Entrada en dos tiempos, **la IA propone / la persona decide**:
  - **Importar:** texto libre (`.md`/`.txt` o pegado) -> `author.draft_project` (Claude) propone un
    **borrador** Pydantic (`ProjectDraft`: título, brief, escenas con planos). Se materializa con el
    nuevo `project.write_spec` (round-trip de `load_project_spec`, idempotente, YAML legible).
  - **Storyboard editable** (renombra *Guion*): editar título/brief y, por escena/plano,
    prompt/beat/duración/voz/caption; **agregar, eliminar y reordenar** escenas. Guardar hace
    `PUT /api/projects/{slug}` que **valida con Pydantic** (422 legible si no cierra).
  - **Parseo único del spec:** `project.spec_from_dict` lo usan tanto el `project.yaml` en disco
    como el PUT (refuerza [D-023]: el contrato es la lógica).
  - **Preservación:** el PUT **fusiona** por id de escena — los campos que la UI no edita (seed,
    `class`, requirements, dialogue, voice_id) se conservan del spec en disco.
  - **Guard de selecciones ([D-022]):** reordenar **no** rompe (el id se conserva); renombrar o
    eliminar una escena **poda** sus entradas huérfanas de `selections.yaml`.
  - **Sin `python-multipart`:** el backend recibe **texto**; la UI lee el archivo client-side
    (FileReader). `ingest.extract_text` queda como helper del motor/CLI para `.md`/`.txt`.
  - **Estado ([D-032]) reconoce "sin proyecto":** `nextStep` guía a *Importar* cuando no hay ninguno.

- **Consecuencias:** Se puede crear y editar un proyecto entero sin tocar YAML. El **hash del caché**
  ([D-013]) no cambia: `write_spec` no toca llaves, solo serializa el spec. La identidad de escena
  (id) pasa a ser el ancla del bucle — por eso el guard.

- **Backlog ya implementado** (post-cierre): personajes con `design:` auto desde el import (#8, la IA
  los propone con su prompt de diseño → habilita el casting) y estilo elegible al importar (#10,
  `GET /api/styles` + selector). **Diferido aún**: `.docx`/`.pdf` (#7); regenerar una escena del
  borrador con la IA (#9); subir imágenes de referencia (#11); renombrar el slug (#12).

## D-034 — Diseño sonoro: SFX + ambiente vía video-to-audio (MMAudio), nativo como premium
**Estado:** Vigente · **Fecha:** 2026-06-06

- **Contexto:** La banda sonora tiene **4 capas**: voces, música, **efectos (SFX)** y **ambiente**.
  Voces (ElevenLabs TTS, por plano, [D-024]) y música (pista global, Sprint 5) ya existen; faltan
  SFX y ambiente. Riesgo a evitar: generar el audio **separado del video descuadra** con la acción
  en pantalla.

- **Investigación (jun-2026, breve).** Dos familias de modelos:
  - **(A) Audio nativo** — el modelo genera imagen+sonido juntos. En fal con **i2v + audio**:
    Veo 3.1 (~$0.40/s c/audio 1080p; Lite $0.05/s 720p), Kling v3 Pro (~$0.168/s), Seedance 2.0
    (estéreo). Integrado y con diálogo/lip-sync, pero **caro** y la indicación de audio va en el
    prompt de generación.
  - **(B) Video-to-audio (V2A)** — toma el clip **ya generado** + un prompt y, **leyendo los frames**,
    genera audio alineado. **MMAudio V2** en fal: $0.001/s, 1–30s, devuelve el clip con el audio
    sincronizado muxeado. (ThinkSound, similar.)
  - Descartados: ElevenLabs Sound Effects (texto→audio **ciego**, no ve el video → es el descuadre
    que queremos evitar); Sora 2 (en deprecación); self-host (choca con [D-017]).

- **Decisión:** **SFX + ambiente con MMAudio V2 (V2A en fal) como ruta por defecto**; audio nativo
  (Veo/Kling) como **premium por `routing.yaml`** (escenas `hero` o con diálogo/lip-sync).
  - **Granularidad** (mapea a [D-028]): **`scene.ambience`** (texto, el *lugar*, por escena) y
    **`shot.sfx`** (texto, la *acción*, por plano), authored en `project.yaml`. El cue de MMAudio por
    plano = `shot.sfx` + `ambience` de su escena.
  - **Sin descuadre y barato:** MMAudio condiciona en los frames del clip real → sincronizado, a
    centavos, **sin tocar** el flujo keyframe→i2v ni el caché. Es un **paso nuevo cacheado** (como la
    VO), llaveado por `video_key + cue`; best-effort (sin cue/clave → se omite).
  - **Jerarquía de mezcla:** voz (TTS) **encima** del audio diegético (MMAudio) **encima** de la
    música. `audio.mux_voiceover` pasa a **mezclar (`amix`)**, no reemplazar (hoy reemplaza → borraría
    el sfx/ambiente). Volúmenes ~ voz 1.0 / diegético 0.6 / música 0.25.
  - **Nativo por routing:** una escena con audio nativo (capability `audio`, ya en `providers.yaml`)
    se va a Veo/Kling y **salta** el paso V2A. Decisión por YAML, sin código ([D-002]/[D-006]).

- **Consecuencias:** Se completa la banda sonora reusando todo (fal, caché, mux, storyboard). El
  diálogo/lip-sync nativo y los **stems separados por capa** en el export ([D-029]) quedan diferidos.
  Cambio real mínimo: `mux_voiceover` mezcla en vez de reemplazar + un adapter `providers/mmaudio`.

- **Cambios (2026-06-06):** el modelo y el costo de MMAudio **no se hardcodean**: viven en config
  (`audio:` en `providers.yaml`, cargado por `config.load_audio` → `Config.audio`), igual que el
  resto de los modelos ([D-002]/[D-020]); ese bloque **no entra al routing**. El costo del paso V2A
  **entra a telemetría** (`SceneRecord.audio_provider`/`audio_cost_usd`; `totals` lo suma y lo expone
  como línea propia en `cost_by_provider`) → el `run_report` cuadra. El model ID `fal-ai/mmaudio-v2`
  sigue **sin validar contra la API real** hasta el smoke pago.

## D-035 — UX del bucle: Configuración fuera del bucle + firma explícita del plan
**Estado:** Vigente · **Fecha:** 2026-06-06

- **Contexto:** En la app (Fase 1.5, [D-031]/[D-032]) las **claves de API** estaban como "paso 1"
  del bucle numerado — pero son *setup* transversal, no un eslabón del loop, y hacían ruido una vez
  cargadas. Y el Storyboard decía "firmá el plan" pero el único botón era "Guardar", que **no
  cambiaba el estado** (el paso nunca prendía su chulo): el Checkpoint #1 de [D-021] no estaba cerrado
  como acto explícito.

- **Decisión:**
  - **Configuración aparte del bucle (#4):** las claves salen de los pasos numerados y pasan a
    **"Configuración"** (un acceso con engranaje, fuera del loop). El bucle queda *Inicio · 1 Importar
    · 2 Storyboard · 3 Elegir · 4 Producir*. Inicio muestra el aviso de claves **solo si faltan**.
  - **Firma del plan (#5):** el Storyboard gana un acto explícito **"Firmar el plan"** — el humano
    *firma* el borrador de la IA (cierra el Checkpoint #1 de [D-021]). Se persiste un **marcador en
    disco** (`storyboard.signed`); **editar sin firmar lo limpia** (el plan cambió → re-firmar).
    `PUT` acepta `sign`; `GET /status` reporta `storyboard.signed`; el estado derivado ([D-032])
    prende el paso 2 al firmar. (`Guardar borrador` queda como acción secundaria.)

- **Consecuencias:** El bucle refleja solo decisiones del flujo; el setup no compite con ellas. La
  "firma" es estado derivado del disco (coherente con [D-032]) y un checkpoint humano real, no un
  guardado silencioso.

## D-036 — Biblioteca global de assets reusables (personajes / símbolos / lugares)
**Estado:** Vigente (en diseño) · **Fecha:** 2026-06-06

- **Contexto:** Hoy los personajes (con `design`/casting, [D-019]) viven **por proyecto**: la cara se
  diseña y elige dentro de `projects/<slug>/` y no se reusa en otro proyecto. Para una marca/serie hace
  falta **crear un asset una vez y reusarlo consistentemente** entre proyectos — y no solo personajes:
  también **símbolos** (logos, íconos) y **lugares** (escenarios recurrentes).

- **Decisión (alcance acordado):** una **biblioteca global** versionada en la raíz del repo
  (p.ej. `library/characters|symbols|places/<nombre>/`) que los proyectos **referencian por nombre**
  (no copian). El asset se diseña/fija una vez (su cara/forma canónica) y se propaga a los keyframes
  igual que el casting actual ([D-019]). Reusa el caché content-addressed ([D-013]); la referencia se
  resuelve a ruta absoluta **solo para I/O** (mismo criterio que [D-034]/`resolve_refs`), manteniendo
  el hash estable.
  - **Diferido / a definir en la implementación:** cómo se nombran y versionan; resolución de refs
    globales vs project-relative; si el cache de assets es global o por-proyecto; UI de gestión.
  - **Alternativa descartada:** "copiar asset de otro proyecto" (sin banco global) — más simple pero
    cada proyecto deriva su copia con el tiempo (inconsistencia, justo lo que se quiere evitar).

- **Consecuencias:** Es un cambio transversal (motor: resolución de refs globales + propagación;
  app: pantalla para crear/elegir assets). Va como sprint propio (ver ROADMAP).

## D-037 — Frame por plano (auto) + cortes cortos (refina D-028)
**Estado:** Vigente (en diseño) · **Fecha:** 2026-06-06

- **Contexto:** [D-028] hizo que el humano elija **un** keyframe por *escena* (= plano 1) y los planos
  2+ autogeneren el suyo; el checkpoint solo muestra el encuadre del plano 1. En la práctica eso se
  percibe como "se reusa una opción para varios planos". Además los planos de ~5s son **largos** para
  formatos de atención corta.

- **Decisión (alcance acordado):** **cada plano autogenera su propio frame** desde su `framing`
  (sin checkpoint humano por plano — "auto por plano"), alimenta su video y se **corta a la duración
  del plano**, con **default corto (~2s)**. El checkpoint humano sigue existiendo a nivel escena/
  casting; no se agrega selección por plano (se mantiene el espíritu de [D-028] de no multiplicar
  checkpoints).
  - **Alternativas descartadas:** selección humana *por plano* (más control, más fricción); dejar todo
    como [D-028] y solo acortar (no resuelve el "frame respectivo por plano").

- **Consecuencias:** Toca `runner._render_shot` (generar el frame de cada plano, incl. el 1, desde su
  framing) y el default/recorte de duración (`trim_to` ya recorta; falta el default ~2s). Va como
  sprint en el ROADMAP del motor.
