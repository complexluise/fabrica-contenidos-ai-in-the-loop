## D-031 — Studio local: app web local sobre el pipeline (FastAPI + Svelte, sin auth)
**Estado:** Vigente · **Fecha:** 2026-06-05

- **Contexto:** El CLI sirve a quien sabe terminal; pero el **equipo** (narradora, diseñadora,
  ingeniero de IA, editora — ver `docs/flujo-de-trabajo.md`) necesita una **UI** para operar los
  checkpoints de [D-021]. Se decidió hacerlo **local** (corre en la máquina del usuario): eso
  **elimina** auth, usuarios, base de datos y nube — *keep it simple*. El estado sigue en disco.

- **Decisión:** Una **app local**: servidor **FastAPI** + UI **Svelte**; el navegador en `localhost`
  es la interfaz. `pipeline studio` levanta el server y abre el browser.
  - **La API es una cáscara fina sobre el motor** (`studio`/`runner`/`export`): el CLI y la API
    llaman a **las mismas funciones** (refuerza [D-023] — el contrato es la lógica, no el shell).
  - **Sin auth / usuarios / Postgres / object storage:** single-user local; el estado vive en
    `projects/`, `runs/`, `cache/` (disco), como hoy.
  - **Jobs async en proceso** (el motor ya es asyncio): cada `cast`/`keyframes`/`render` es una
    **tarea con id**; el progreso se transmite **en vivo por SSE**, reutilizando el logging de
    [D-027]. **Paralelo entre jobs** con un semáforo de concurrencia (**Fase 2**). **Planos
    concurrentes dentro del render: diferido** (Fase 3).
  - **API keys desde la UI** (pantalla de Ajustes): se escriben a un config local; se invalida el
    `lru_cache` de `settings` al guardar.
  - **Las pantallas = los checkpoints ([D-021]):** proyecto/guion · **picker** (reemplaza la hoja de
    contactos) · render (progreso vivo) · export · ajustes.

- **Consecuencias:**
  - Es un **track nuevo** (la app), con su **propio ROADMAP** (`app/ROADMAP.md`). El pipeline
    (motor) **no cambia su contrato**; gana endpoints. Backend en `src/pipeline/server/`, UI en `app/`.
  - **Reutiliza todo:** `studio`/`runner`/`export` + el logging + el caché + el bundle. La hoja de
    contactos HTML ([D-022]) era el **prototipo** del picker.
  - **AI-in-the-Loop hecho producto:** la UI **es** el loop — donde la persona decide y firma.

- **Quitado (navaja de Ockham, por ser local):** auth, multi-tenancy, Postgres, S3, deploy/ops.

- **Cambios:** Nuevo `src/pipeline/server/` (FastAPI + job manager) + `app/` (Svelte) + subcomando
  `pipeline studio`. ROADMAP propio en `app/ROADMAP.md`.

## D-032 — Estado del bucle **derivado del disco** (fuente única para server y UI)
**Estado:** Vigente · **Fecha:** 2026-06-06

- **Contexto:** La UI (Fase 1.5) necesita decir *dónde está parada* la persona (keys · casting ·
  encuadres elegidos · render · export) y *cuál es el siguiente paso*. La tentación es guardar un
  "estado de avance" propio; pero eso se desincroniza del disco (el estado real ya vive en
  `casting.yaml`/`selections.yaml`/`runs/`/`export/`, [D-013]/[D-022]).

- **Decisión:** El estado del bucle se **deriva** del disco, no se persiste aparte. Una **fuente
  única**: el endpoint `GET /api/projects/{slug}/status` lee los archivos del proyecto y devuelve
  el avance; el front tiene **una sola** función de recomendación (`nextStep`) que traduce ese
  estado en el próximo paso. Sin estado duplicado, sin máquina de estados explícita.

- **Consecuencias:** Imposible desincronizar (si borrás `selections.yaml`, el estado lo refleja).
  Barato (lectura de archivos). El front no decide el flujo: lo lee. Editar el spec a mano sigue
  siendo válido — el estado se recalcula solo.

## D-033 — Entrada desde la app: importar texto -> storyboard editable (Checkpoint #1 hecho UI)
**Estado:** Vigente · **Fecha:** 2026-06-06

- **Contexto:** Arrancar un proyecto exigía escribir `project.yaml` a mano — barrera para el equipo
  no-técnico ([D-031]). El Checkpoint #1/#2 de [D-021] (la IA descompone, la persona firma) no
  estaba en la interfaz. Faltaba además el **escritor** del spec: el motor sabía leer `project.yaml`
  pero no escribirlo.

- **Decisión:** Entrada en dos tiempos, **la IA propone / la persona decide**:
  - **Importar:** texto libre (`.md`/`.txt` o pegado) -> `author.draft_project` (Claude) propone un
    **borrador** Pydantic (`ProjectDraft`: título, brief, escenas con planos). Se materializa con el
    nuevo `project.write_spec` (round-trip de `load_project_spec`, idempotente, YAML legible).
  - **Storyboard editable** (renombra *Guion*): editar título/brief y, por escena/plano,
    prompt/beat/duración/voz/caption; **agregar, eliminar y reordenar** escenas. Guardar hace
    `PUT /api/projects/{slug}` que **valida con Pydantic** (422 legible si no cierra).
  - **Parseo único del spec:** `project.spec_from_dict` lo usan tanto el `project.yaml` en disco
    como el PUT (refuerza [D-023]: el contrato es la lógica).
  - **Preservación:** el PUT **fusiona** por id de escena — los campos que la UI no edita (seed,
    `class`, requirements, dialogue, voice_id) se conservan del spec en disco.
  - **Guard de selecciones ([D-022]):** reordenar **no** rompe (el id se conserva); renombrar o
    eliminar una escena **poda** sus entradas huérfanas de `selections.yaml`.
  - **Sin `python-multipart`:** el backend recibe **texto**; la UI lee el archivo client-side
    (FileReader). `ingest.extract_text` queda como helper del motor/CLI para `.md`/`.txt`.
  - **Estado ([D-032]) reconoce "sin proyecto":** `nextStep` guía a *Importar* cuando no hay ninguno.

- **Consecuencias:** Se puede crear y editar un proyecto entero sin tocar YAML. El **hash del caché**
  ([D-013]) no cambia: `write_spec` no toca llaves, solo serializa el spec. La identidad de escena
  (id) pasa a ser el ancla del bucle — por eso el guard.

- **Backlog ya implementado** (post-cierre): personajes con `design:` auto desde el import (#8, la IA
  los propone con su prompt de diseño → habilita el casting) y estilo elegible al importar (#10,
  `GET /api/styles` + selector). **Diferido aún**: `.docx`/`.pdf` (#7); regenerar una escena del
  borrador con la IA (#9); subir imágenes de referencia (#11); renombrar el slug (#12).

## D-034 — Diseño sonoro: SFX + ambiente vía video-to-audio (MMAudio), nativo como premium
**Estado:** Vigente · **Fecha:** 2026-06-06

- **Contexto:** La banda sonora tiene **4 capas**: voces, música, **efectos (SFX)** y **ambiente**.
  Voces (ElevenLabs TTS, por plano, [D-024]) y música (pista global, Sprint 5) ya existen; faltan
  SFX y ambiente. Riesgo a evitar: generar el audio **separado del video descuadra** con la acción
  en pantalla.

- **Investigación (jun-2026, breve).** Dos familias de modelos:
  - **(A) Audio nativo** — el modelo genera imagen+sonido juntos. En fal con **i2v + audio**:
    Veo 3.1 (~$0.40/s c/audio 1080p; Lite $0.05/s 720p), Kling v3 Pro (~$0.168/s), Seedance 2.0
    (estéreo). Integrado y con diálogo/lip-sync, pero **caro** y la indicación de audio va en el
    prompt de generación.
  - **(B) Video-to-audio (V2A)** — toma el clip **ya generado** + un prompt y, **leyendo los frames**,
    genera audio alineado. **MMAudio V2** en fal: $0.001/s, 1–30s, devuelve el clip con el audio
    sincronizado muxeado. (ThinkSound, similar.)
  - Descartados: ElevenLabs Sound Effects (texto→audio **ciego**, no ve el video → es el descuadre
    que queremos evitar); Sora 2 (en deprecación); self-host (choca con [D-017]).

- **Decisión:** **SFX + ambiente con MMAudio V2 (V2A en fal) como ruta por defecto**; audio nativo
  (Veo/Kling) como **premium por `routing.yaml`** (escenas `hero` o con diálogo/lip-sync).
  - **Granularidad** (mapea a [D-028]): **`scene.ambience`** (texto, el *lugar*, por escena) y
    **`shot.sfx`** (texto, la *acción*, por plano), authored en `project.yaml`. El cue de MMAudio por
    plano = `shot.sfx` + `ambience` de su escena.
  - **Sin descuadre y barato:** MMAudio condiciona en los frames del clip real → sincronizado, a
    centavos, **sin tocar** el flujo keyframe→i2v ni el caché. Es un **paso nuevo cacheado** (como la
    VO), llaveado por `video_key + cue`; best-effort (sin cue/clave → se omite).
  - **Jerarquía de mezcla:** voz (TTS) **encima** del audio diegético (MMAudio) **encima** de la
    música. `audio.mux_voiceover` pasa a **mezclar (`amix`)**, no reemplazar (hoy reemplaza → borraría
    el sfx/ambiente). Volúmenes ~ voz 1.0 / diegético 0.6 / música 0.25.
  - **Nativo por routing:** una escena con audio nativo (capability `audio`, ya en `providers.yaml`)
    se va a Veo/Kling y **salta** el paso V2A. Decisión por YAML, sin código ([D-002]/[D-006]).

- **Consecuencias:** Se completa la banda sonora reusando todo (fal, caché, mux, storyboard). El
  diálogo/lip-sync nativo y los **stems separados por capa** en el export ([D-029]) quedan diferidos.
  Cambio real mínimo: `mux_voiceover` mezcla en vez de reemplazar + un adapter `providers/mmaudio`.
