## D-021 — "AI in the Loop": diseñar primero el flujo humano
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** El pipeline nació **AI-autónomo** (un `pipeline run` hace todo; el humano solo
  toca el brief). Pero la generación de video es una tarea creativa con decisiones que el humano
  debe poseer. Invertimos el encuadre: no "human in the loop" (humano que aprueba lo que la IA
  decide), sino **"AI in the loop"** — se diseña primero el flujo de decisiones del humano y la
  IA rellena los huecos (genera y asiste).

- **Decisión:** El humano dirige; la IA **genera N candidatos y asiste**. Checkpoints humanos:

  | # | Checkpoint | Quién decide | Rol de la IA | Capa |
  |---|---|---|---|---|
  | 1 | Guion | Humano | redacta/expande | L1 |
  | 2 | Shot list / metadata | Humano edita/aprueba | descompone, clasifica, propone duración/cámara | L1+L2 |
  | 3 | **Casting + look-dev** (personaje + estilo) | Humano elige | N variantes de personaje/estilo | banco ([D-019]) + style ([D-003]) |
  | 4 | **Keyframe** (best-of-N) | Humano elige | N candidatos + ranking | L3 |
  | 5 | **Video** (best-of-N en hero) | Humano elige/aprueba | genera + auto-juzga | L4/L5/L6 |
  | 6 | **Corte / edición** | Humano | sugiere orden/ritmo/transiciones, auto-captions | L7 |
  | 7 | Sonido (opcional) | Humano | genera/sugiere voz y música | nuevo |
  | 8 | Publicación (opcional) | Humano go/no-go | caption/hashtags/thumbnail por plataforma | L8 |

  Los **tres puntos núcleo** definidos por el usuario son **2 (shot list)+1 (guion)**, **4
  (keyframe)** y **5 (video)**. Se añaden como explícitos **3 (casting/look-dev)** —la base de
  toda la consistencia, todo lo hereda— y **6 (corte)** —donde los clips se vuelven un video—.

- **Consecuencias:**
  - **El Gate cambia de juez a asistente:** pre-puntúa y **ordena los N candidatos** para que el
    humano elija más rápido (mejor primero, marca artefactos), no un pass/fail autónomo. Encaja
    con el gate suave ([D-018]); `gate_score` se vuelve *ranking de candidatos*.
  - **El pipeline pasa de autónomo a por etapas, reanudable, con sets de candidatos + selección
    persistida:** cada checkpoint → IA genera N (cacheados) → gate ordena → humano selecciona →
    la selección se guarda en el manifiesto/spec → la siguiente etapa la consume.
  - Aprovecha el sustrato ya construido: candidatos = artefactos por hash ([D-013]); el "reroll
    por seed" ([D-014]) es el embrión de best-of-N; los runs-manifiesto ([D-015]) registran la
    selección. Habilita durabilidad futura ([D-009]) y una UI.

- **Cambios:** Principio rector nuevo; reencuadra el rol del Gate ([D-016]/[D-018]) y motiva un
  rediseño del pipeline en checkpoints (best-of-N + selección), empezando por el keyframe (#4).

## D-022 — Interacción humano-IA: subcomandos con estado + hoja de contactos
**Estado:** Revisada (flexibilizada por [D-025]) · **Fecha:** 2026-06-02

- **Contexto:** Implementar los checkpoints de [D-021] exige un modelo de interacción. El factor
  que manda son los **tiempos largos** (keyframe ~30-60s, video 1-3 min, ×N en best-of-N): el
  humano **no puede quedarse bloqueado** esperando. El patrón debe ser *disparo → me voy → vuelvo
  → reviso → selecciono → disparo lo siguiente*. Opciones consideradas: (A) subcomandos pasando el
  resultado a mano, (B) subcomandos con el estado guardado en el proyecto, (C) hoja de contactos
  HTML, (D) mini-web (Gradio/Streamlit), (E) background + aviso.

- **Decisión:** Combinar **B + C + E** (keep it simple but functional):
  - **B** — subcomandos donde **el proyecto recuerda la selección** (resumible, sin malabares de
    params): `keyframes` → `pick` → `render`. La selección se persiste en `selections.yaml`.
  - **C** — al generar candidatos, se escribe una **hoja de contactos HTML** que se auto-abre
    (grid de candidatos etiquetados por escena/índice). Ideal para keyframes (imágenes).
  - **E** — la generación es **no bloqueante**; la hoja de contactos al abrirse actúa de "listo".
  - **Asimetría keyframe ≠ video:** keyframe = best-of-N + humano elige (imágenes, fácil de ver
    en grid); **video** = IA elige el mejor (Ensemble + gate-ranking) y el humano **vetea/rerollea**
    (no best-of-N completo, porque revisar N videos es caro).
  - **(D) mini-web** queda como evolución natural cuando duela, no ahora.

- **Consecuencias:** El pipeline gana un **modo por etapas** (`keyframes`/`pick`/`render`) además
  del `run` autónomo, que se mantiene. Aprovecha el caché ([D-013]) y los runs-manifiesto
  ([D-015]). El Gate actúa de **ranking** de candidatos ([D-021]).

- **Cambios:** Primera implementación del checkpoint de keyframe ([D-021] #4).

## D-023 — Skills como capa de discoverability para agentes
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** El pipeline ya implementa el patrón **catalog → renderer** ([D-013]): el catálogo
  es `project.yaml`, el renderer es `runner.py` + módulos en `src/pipeline/`. Pero un agente
  (opencode, claude code) que opera sobre el pipeline hoy tiene que descubrir las herramientas
  leyendo el código. Una referencia de proyecto paralelo
  (`docs/SPEC-organic-illustration-pipeline.md`) mostró una separación más limpia: catálogo
  estructurado, script determinista, agente que sigue reglas de autoría. Falta en este repo la
  **capa de skills** que vuelva esas semi-tools descubribles para un agente.

- **Decisión:** Formalizar una capa de **skills** en `skills/<nombre>/SKILL.md`. Cada skill es:
  - **Una guía en markdown** (no código nuevo, no reescritura).
  - **Apunta al contrato CLI estable** (`pipeline keyframes`, `pick`, `render`, `run`, `cast`…),
    **no a clases internas** (`KeyframeGenerator`, `classify`, `project.load_project_spec`).
    El CLI es la superficie pública versionada; el grafo de objetos de `src/pipeline/` es interno
    y refactorizable. Acoplar la prosa de una skill a una clase Python es frágil; acoplarla a un
    subcomando es un contrato que podemos defender (ver gestión de riesgo abajo).
  - **Sin `scripts/` auxiliares** por ahora — el pipeline CLI es suficiente.
  - **Versionada con el código** en el repo (visible para todo el equipo, no solo en la
    config del agente).
  - **Con un smoke de contrato en CI** (gestión del riesgo de drift, ver abajo).

- **Consecuencias:**
  - **No se reimplementa lógica.** Las skills apuntan al CLI, no duplican `src/pipeline/`.
  - **El equipo no-técnico puede authorar skills** para sus propios flujos (un marketero puede
    crear `skill: brief-a-videos`, un editor `skill: reordenar-cortes`).
  - **El primer batch (3 skills) cubre el flujo típico**: `author-project`, `bank-casting`,
    `keyframe-best-of-n`. Más skills se agregan por sprint cuando aparece la necesidad.

- **Gestión del riesgo (drift silencioso):** una skill es prosa que codifica conocimiento del
  CLI; si un flag o un formato de salida cambian, la skill **queda obsoleta sin que nada falle**.
  Es el mismo riesgo que el propio SPEC de referencia documenta como *"Known drift"* (§10: el
  docstring dice "Imagen 3" pero el código llama a Imagen 4) y como F10 (hash de `STYLE_BASE`).
  Dos contramedidas:
  - **Acoplar al contrato CLI, no a internals** (ver Decisión). El CLI cambia con cadencia de
    release y de forma visible; las clases internas cambian en cualquier refactor.
  - **Smoke de contrato en CI:** cada `SKILL.md` declara al final un bloque
    `<!-- smoke: ... -->` con las invocaciones mínimas que menciona (p.ej.
    `pipeline keyframes --list`, `pipeline --help`). CI las ejecuta en modo *no-op* (sin llamar a
    modelos); si un subcomando o flag referenciado desaparece, **el test grita** en vez de que la
    skill se pudra callada. Es el equivalente, para la capa de skills, del aviso de drift de F10.

- **Lo que esto le exige al CLI (para ser usable por una IA tanto como por un humano):** las
  propiedades que hacen el CLI navegable para un agente son las mismas que mejoran la DX humana,
  no compiten:
  - **Auto-descriptivo sin gastar:** `--list`/`--help` y listar ids/escenas **sin llamar a
    modelos** (modo *list* puro), para que el smoke y el agente exploren gratis.
  - **Estado legible por máquina:** la selección y el progreso viven en `selections.yaml` /
    manifiesto ([D-015]/[D-022]), no en prosa que el agente tenga que parsear.
  - **Errores que nombran el problema + exit codes claros** (estilo F1–F9 del SPEC de ref.):
    "falta env var X en `.env`", "id desconocido, usa `--list`".
  - **Idempotencia y fallo acotado:** re-correr sobrescribe; un fallo en una escena no aborta el
    batch (imprescindible cuando un agente dispara N candidatos sin supervisión).
  - **`--dry-run`/`--validate`:** el agente verifica sin quemar generaciones.

- **Cambios:** Primera implementación en este sprint; ver `skills/`. El smoke de contrato se
  suma a la suite de CI. Refuerza el contrato CLI de [D-022] como **superficie pública**.

## D-024 — Voz en off por escena con ElevenLabs (TTS), opcional
**Estado:** Vigente · **Fecha:** 2026-06-03

- **Contexto:** El audio del Sprint 5 era solo música de fondo. Falta **narración**. No todos los
  videos la necesitan: es **por-escena y opcional**. Alineado a [[prefer-apis-over-heavy-libs]]:
  TTS por API (ElevenLabs), sin modelos de voz locales.

- **Decisión:**
  - **`voiceover:` (texto) opcional por escena.** Si está, se sintetiza con ElevenLabs y se mezcla
    como pista de audio del clip.
  - **Voz: default de proyecto + override por escena.** `voice_id` a nivel `project.yaml`; `voice_id`
    por escena lo sobreescribe; si no hay ninguno, una **voz multilingüe stock** (modelo
    `eleven_multilingual_v2`, cubre español). Resolución pura y testeada (`resolve_voice`).
  - **Adapter con `httpx` crudo, no la SDK** (`providers/elevenlabs_tts.py`): una sola llamada
    HTTP, sin dependencia nueva (httpx ya está en el core). Errores HTTP traducidos a mensajes que
    **nombran el problema** (401 key, 402 sin crédito, 404 voz, 429 rate limit) — estilo [D-023].
  - **Caption se autocompleta de la VO.** Si hay `voiceover` y no `caption`, el texto en pantalla
    sale del de la voz (sin whisper). `effective_caption` puro y testeado.
  - **Música *ducked* bajo la narración.** Cuando una VO se mezcló de verdad, la música baja
    (volumen 0.25) vía `amix`; si no, queda a volumen pleno (comportamiento Sprint 5). El audio se
    **normaliza** antes de concatenar (pista aac uniforme, silencio si no hay) para soportar clips
    con y sin VO en el mismo video.
  - **VO cacheada content-addressed** (`hash(texto+voz+modelo)`, [D-013]): re-correr no re-sintetiza.
  - **Best-effort (robustez [D-018]/Sprint 5):** si falta la key, no hay crédito, o falla la
    síntesis, la escena **no se pierde** — sale con música + caption escrita.

- **Consecuencias:** Nuevo `Scene.voiceover`/`Scene.voice_id`, `ProjectSpec.voice_id`,
  `settings.elevenlabs_api_key`, módulo `audio.py` (lógica pura + mezcla ffmpeg) y el adapter. El
  ensamblaje pasa de *reemplazar* el audio con música a *mezclarla por debajo* (cambio en
  `concat_clips`).

- **Cambios:** Implementado en Sprint 6. La validación PAGA end-to-end (audio real mezclado) queda
  **bloqueada por el crédito de la cuenta ElevenLabs** (la API responde 402); el adapter quedó
  validado hasta auth+formato (402, no 401). El resto del core va con tests.

## D-025 — `render`/selección admiten el artefacto directo por flag (no solo el ciclo de candidatos)
**Estado:** Vigente · **Fecha:** 2026-06-04

- **Contexto:** El flujo de [D-022] (`keyframes → pick → render`, `cast → pick-cast`) asume que el
  humano **siempre elige entre N candidatos generados**. Pero hay casos donde la imagen final **ya
  existe** —viene de afuera, de una corrida previa, o ya está decidida—: no hay nada que *elegir*,
  solo que *tomar*. Forzar el ciclo generación+`pick` es burocracia y **bloquea casos legítimos**.

- **Decisión:** `render` (y el casting) aceptan **inyectar el artefacto directo por flag**,
  salteándose la generación de candidatos y el `pick`:

  ```bash
  pipeline render <slug> --keyframe s1=ruta --keyframe s2=ruta   # repetible, por escena
  ```

  El flag **toma precedencia** sobre `selections.yaml` para esa escena; el resto de escenas sigue
  leyendo la selección persistida. Análogo para el casting (fijar una cara directa). Se usa
  **flag** (no carpeta-convención ni autodetección) **a propósito**: los artefactos son de tipos
  distintos (keyframe / cara de personaje / otros) y el flag hace **explícito qué tipo** se inyecta.

- **Consecuencias:** El modo persistido de [D-022] (opción B) sigue siendo **un** camino, no el
  único — por eso [D-022] pasa a **Revisada (flexibilizada)**. Mantiene el principio [D-021] (el
  humano dirige): puede inyectar su decisión sin pasar por la generación de candidatos.
  `run_project` ya acepta `keyframe_overrides` → el flag solo lo **cablea desde el CLI**. El
  contrato del CLI gana flags → actualizar `skills/*` + test de contrato ([D-023]). Alimentado por
  [D-026]: el nombre legible es lo que el humano pasa al flag.

- **Cambios:** Flexibiliza [D-022] (su acople secuencial deja de ser obligatorio).

## D-026 — Artefactos con nombre semántico legible (humano-first); el hash queda como verdad del caché
**Estado:** Vigente · **Fecha:** 2026-06-04

- **Contexto:** [D-021] (AI-in-the-Loop) implica que el humano **mira, elige y manipula los archivos
  a mano** —y [D-025] lo refuerza: para "tomar" una imagen, el humano necesita poder **nombrarla y
  encontrarla**. Hoy los artefactos se guardan content-addressed (`sha256[:16].png`) y la hoja de
  contactos los etiqueta solo por `escena/índice`: abrir `projects/<slug>/cache/keyframes/` es una
  **sopa de hashes**, hostil para un humano.

- **Decisión:** Exponer un **nombre semántico legible** como capa visible del artefacto
  (`<escena>_<slug-descriptivo>_<idx>.png` para keyframes, `<personaje>_cara_<idx>.png` para
  casting). El **hash no se elimina**: sigue siendo la clave del caché y la procedencia
  ([D-013]/[D-015]) —romperlo invalidaría el caché content-addressed—. El nombre legible es un
  **alias/copia humano-facing** (p.ej. en `projects/<slug>/keyframes/` y en la hoja de contactos)
  que el humano abre y que el flag de [D-025] acepta. El slug descriptivo se deriva con **Claude
  (`ANTHROPIC_API_KEY`, el modelo más barato — Haiku)** desde el prompt/escena; **sin key**, cae a
  un slug del `id` de la escena/personaje.

- **Consecuencias:** El humano puede tomar archivos a mano (alimenta [D-025]). El caché
  content-addressed queda **intacto**: la legibilidad es una capa encima, no un reemplazo. Costo
  marginal: una llamada barata de naming por artefacto, **cacheable** (re-correr = $0). Encaja con
  [D-017] (preferir una API barata antes que una heurística local frágil).

- **Cambios:** Nueva capa de naming; toca `studio.py` (gen/cast), `contact_sheet` (etiquetas) y el
  CLI (el flag de [D-025] acepta el nombre legible).

## D-027 — Logging detallado (L9): módulo `logging` + RichHandler a stderr + `run.log` por corrida
**Estado:** Vigente · **Fecha:** 2026-06-05

- **Contexto:** Durante `render`/`run`/`cast` el pipeline era **mudo por minutos** (la generación
  paga es lenta) y los **errores por escena eran invisibles**: el `except` de `run_project` solo
  llamaba a `telemetry.record_failure` → quedaban en `run_report.json`, nunca en consola. El humano
  no podía ver el avance (qué imagen se generó, cache hit/miss, costo) ni los fallos en vivo.

- **Decisión:** Las capas de librería (`runner`, `studio`, …) usan `logging.getLogger(__name__)`;
  el CLI configura **un `RichHandler` a stderr** (`logging_setup.setup_logging`). Claves:
  - **stdout queda limpio** para el resumen final + paths → respeta el CLI como **contrato
    dual-audiencia** ([D-023]): un agente parsea stdout, el humano ve el progreso en stderr.
  - **`-v/--verbose`** (global, en el callback) → DEBUG con tracebacks; por defecto **INFO**
    (progreso + errores en una línea limpia).
  - **`run.log` por corrida:** `run_project` adjunta un `FileHandler` DEBUG en
    `runs/<run_id>/run.log` (detalle completo + tracebacks) y lo quita al cerrar.
  - **Errores visibles + robustez:** el fallo por escena/candidato se **loguea a consola** (ERROR)
    además de registrarse en el report; `cast`/`keyframes` ganan **try/except por candidato** (uno
    que falla no tira los demás), igual que `run_project` por escena.

- **Consecuencias:** Visibilidad en vivo del avance y de los errores; `run.log` para depurar las
  pagas largas después del hecho. `markup=False` en el handler (así `[s2]` es texto, no estilo
  rich). **No es core test-first** (I/O/presentación) → se valida con smoke; la infra
  (`setup_logging`/`add_run_logfile`) se probó sin API.

- **Cambios:** Nuevo `logging_setup.py`; `runner`/`studio` instrumentados; `-v` en el callback del
  CLI. Avanza la capa L9 hacia la observabilidad del Sprint 7 (dashboard) sin adelantarla.

## D-028 — La escena compone planos (`shots`): plano 1 = keyframe elegido; los demás autogenerados
**Estado:** Vigente · **Fecha:** 2026-06-05

- **Contexto:** Un solo plano de N segundos por escena **aburre**: un beat narrativo necesita 2-3
  planos (un montaje: wide → close → detalle). Pero el humano **no quiere multiplicar la
  selección** ("con elegir el keyframe es suficiente"). Hasta hoy `Scene` confundía tres papeles
  —beat narrativo, plano visual y unidad de audio— en un solo objeto. Esto realiza el **checkpoint
  #2 "shot list"** de [D-021] a grano fino, sin agregar fricción al humano.

- **Decisión:** La escena (beat) gana `shots: list[Shot]`; el **plano** pasa a ser la **unidad
  atómica** (keyframe + clip + audio). Reglas:
  - **`Shot` = `{framing, duration_s, seed, voiceover, caption, keyframe}`.** El prompt efectivo del
    plano = `scene.prompt` (base: setting + personajes) **+ `, ` + `framing`** → envuelto por el
    style template. El plano **hereda** personajes/estilo de la escena; **varía** encuadre + audio.
  - **El plano 1 ES el keyframe elegido.** El checkpoint de keyframe **no cambia**: sigue siendo
    **por escena**, best-of-N, y apunta al **plano 1**. `keyframes`/`pick`/`selections.yaml` quedan
    **scene-addressed** (no por plano) → no se multiplica la selección.
  - **Los planos 2+ se generan "de un tiro", sin validación:** keyframe autogenerado (n=1, cacheado)
    desde refs + `scene.prompt + framing`; cada uno con su clip i2v + su audio. El humano no los
    elige; si uno no gusta, **sube su `seed` o edita su `framing`** y re-renderiza (solo ese plano,
    caché [D-013]).
  - **Audio por plano:** `voiceover`/`caption` viven en el `Shot`. El ensamblaje vuelve a ser de
    **un solo nivel** (concat de planos, cada uno con su audio ya mezclado).
  - **Compatibilidad:** escena **sin** `shots:` = **1 plano implícito** (usa `scene.prompt`/
    `duration_s`/`voiceover`/`caption`/`seed`) → comportamiento actual intacto (lego_mix, etc.).

- **Consecuencias:**
  - La **granularidad de iteración** pasa de escena a **plano** → [D-014] queda **Revisada**:
    caché/reroll por plano (`seed` por `Shot`); la escena agrupa planos.
  - El contrato humano-interactivo (`keyframes`/`pick`/`render`/`--keyframe`) sigue **por escena**;
    los planos son **expansión interna** en el `render`. Mínima fricción.
  - Realiza el checkpoint #2 (shot list) de [D-021] de forma **manual** (el humano escribe los
    `framing`); el **auto shot-list** (Claude descompone el beat) queda **diferido**.
  - **Riesgo:** duración mínima del provider (Kling/Seedance ~5s). Si no respeta planos cortos, se
    genera al mínimo y se **recorta con ffmpeg** a `duration_s` (conservador: si la respeta, no
    recorta).

- **Cambios:** Nuevo `Shot` en `contracts.py`; `Scene.shots` + `effective_shots` en `project.py`;
  `runner` expande planos; `assemble` concatena por plano. `transition` sigue **diferido** (metadato;
  corte duro). Auto shot-list diferido.

## D-029 — Export bundle para edición humana (handoff a la editora)
**Estado:** Vigente · **Fecha:** 2026-06-05

- **Contexto:** El `final.mp4` auto-armado **no alcanza**: el video real necesita **edición humana**
  (ritmo, timing de voz, tipografía). AI-in-the-Loop hasta el final → el pipeline deja de pretender
  el corte definitivo y entrega **materia prima limpia + un brief**; la editora hace el corte. El
  `final.mp4` se degrada a **"rough cut" de referencia**. Es la capa **L8 (delivery)** bien hecha.

- **Decisión:** Un comando **`pipeline export <slug>`** arma `projects/<slug>/export/` desde el
  **último run**:
  - **`media/`** — todo el material crudo junto, emparejado por nombre: el **video limpio** por
    plano (sin caption ni VO quemada, recortado a `duration_s`) + su **voz** (`.mp3` si se generó) +
    la **música**. Numeración global `NN_<shot_id>` = orden del timeline (`02_s2.mp4`/`02_s2.mp3`).
  - **`frames/`** — un **keyframe** por plano (mismo nombre), referencia visual.
  - **`rough_cut.mp4`** — el `final.mp4` del run (con voz/caption quemada) como **referencia** de
    orden/ritmo, no el definitivo.
  - **`subtitulos.srt`** — subtítulos **sincronizados** al timeline (texto de voz por plano, timeado
    por la suma de duraciones).
  - **`guion.md`** — **onboarding** en markdown: *primero lo importante* (cómo está organizado + la
    **tabla de planos en orden**: `# · archivo · beat · dur · qué se ve · voz · texto · transición`),
    *luego las definiciones* (plano, beat, keyframe, rough cut…) en lenguaje claro y corto.

- **Consecuencias:**
  - Hereda el **naming legible** ([D-026]) y la **expansión de planos** ([D-028]); el bundle es
    100% human-facing (sin manifest/telemetry/hashes).
  - **Clips limpios** = del **caché** (output crudo del provider) recortados a la duración del plano;
    NO los `captioned`/`voiced` (esos solo alimentan el `rough_cut`). La editora controla texto y voz.
  - El comando es **read-only** sobre el run y **decoupled** del render (re-ejecutable). Lee el
    **manifest** del último run (orden + `video_key`) + el **spec** (texto/duraciones/framings).
  - El **`.srt`**, la **numeración** y la **tabla** del guion son **lógica pura** → core testeable;
    la copia de archivos es I/O (smoke).

- **Quitado (navaja de Ockham):** `project.yaml`/docs fuente (a la editora no le interesan),

## D-030 — `guion.md` como documento completo (script) + conversor md→docx (frontmatter)
**Estado:** Vigente · **Fecha:** 2026-06-05

- **Contexto:** El `guion.md` de [D-029] era una **tabla técnica** para la editora; no contaba la
  historia. Pero la idea del export es que **alguien SIN contexto, solo leyendo, entienda el
  proyecto entero**. Faltaba el **guion** en sí. Además se quiere el documento en **`.docx`** (el
  formato que circula con gente no-técnica). Ya existía un generador de `.docx` con buen estilo
  (`docx` JS, paleta, portada, tablas, notas, footer) pero **hardcodeado** a un contenido.

- **Decisión:**
  - **`render_guion` produce un documento autoexplicativo**, en orden: **frontmatter**
    (`title`/`subtitle`/`footer`) → **sinopsis** (del brief, lenguaje claro) → **personajes** →
    **el guion beat-por-beat** (descripción de la escena + el diálogo/VO por plano, legible como
    libreto) → **desglose por plano** (la tabla, para edición) → **cómo está organizado +
    definiciones**. La descripción de cada escena sale de `scene.prompt` (spec).
  - **Conversor genérico md→docx** en `src/md_to_docs/` (proyecto Node con **pnpm**): lee
    **cualquier `.md`**, toma metadata del **frontmatter** (`gray-matter`), parsea con `marked` y
    mapea a `docx` **reutilizando el sistema de estilo** (paleta por **default**, configurable por
    frontmatter). Soporta headings, párrafos (bold/italic), listas (bullet/numbered), tablas GFM,
    blockquote (→ nota) y `---` (→ divider). CLI: `node convert.js <in.md> [out.docx]`.
  - **`pipeline export` invoca el conversor best-effort:** si hay `node`/`pnpm`, deja `guion.docx`
    en el bundle; si no, **solo `guion.md`** (no rompe). Python **no depende** de Node.

- **Consecuencias:**
  - El bundle ([D-029]) gana un documento legible para **cualquiera** + su `.docx`.
  - El conversor es **reutilizable** para cualquier `.md` del repo (specs, planes), no solo el guion.
  - La generación de markdown sigue siendo **lógica pura** (core testeable); el conversor JS se
    valida con **smoke**. El frontmatter es el **contrato de metadata** (title/subtitle/footer/accent).

- **Cambios:** `render_guion` ampliado (+frontmatter); nuevo `src/md_to_docs/` (`package.json` pnpm,
  `convert.js`); `export_bundle` con invocación best-effort a node.
