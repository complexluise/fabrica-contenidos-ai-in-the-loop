## D-011 — Secretos con pydantic-settings (.env)
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** API keys (fal, Anthropic, Google) fuera del repo, un solo punto de verdad.
- **Decisión:** `pipeline.settings.Settings` (pydantic-settings v2) lee `.env` y env vars reales.
  `.env` gitignored; `.env.example` como plantilla. Las keys se pasan explícitas a cada SDK.
- **Consecuencias:** Nada de `os.environ` suelto; error claro cuando falta una key.

## D-012 — TDD selectivo, test-first del core crítico
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** Calidad sin saturar de pruebas.
- **Decisión:** **Test primero** (red-green-refactor) y **solo** del core crítico: routing/
  estrategias, gate (fusión/umbral), telemetría, contracts, hashing/caché. APIs/ffmpeg/prompts
  se validan con smoke run.
- **Consecuencias:** Suite pequeña y significativa (>60 tests). (Preferencia del usuario.)

## D-013 — Modelo de proyecto: spec + caché content-addressed + runs-manifiesto
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** Muchas iteraciones; el problema dominante es el **costo de iterar**, no el orden.
- **Decisión:** El proyecto es un `project.yaml` declarativo; cada paso se llavea por **hash de
  inputs** y se cachea; un run es un **manifiesto** (config resuelta + punteros) + el render.
- **Consecuencias:** Re-correr sin cambios cuesta $0; comparar runs = diff de manifiestos.

## D-014 — Granularidad de iteración = escena; seed determinista + reroll
**Estado:** Revisada (grano fino → plano, por [D-028]) · **Fecha:** 2026-06-02

- **Contexto:** ¿Qué se regenera al iterar? (video completo / escena / paso).
- **Decisión:** **Escena**. Seed **determinista** por defecto (re-correr = cache hit); subir el
  `seed` de una escena = **reroll** (cache miss solo en esa escena).
- **Consecuencias:** Cambias prompt o rerolleas una escena sin re-pagar el resto.
- **Cambios ([D-028]):** Con la escena compuesta de **planos** (`shots`), el grano fino de
  iteración pasa a ser el **plano**: el `seed` vive en el `Shot` y rerollear un plano regenera solo
  ese (caché). La escena queda como **agrupador** (beat); el plano es la unidad keyframe/clip/caché.

## D-015 — Caché a nivel proyecto + procedencia con sidecar
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** El caché debe compartirse entre runs del mismo proyecto.
- **Decisión:** Caché en `projects/<slug>/cache/` (no global, no por run). Un **sidecar
  `.meta.json`** guarda quién generó cada artefacto y sus `gate_scores`.
- **Consecuencias:** Un run nuevo reusa lo anterior; en cache hit se preserva la procedencia
  (provider real, no "cache") y los scores del gate.

## D-016 — Quality Gate "duro": señales enchufables + fusión ponderada
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** Endurecer el gate sin acoplarlo a un solo método.
- **Decisión:** Paquete `gate/` con **señales** (VLM, identidad, y opcionales CLIP/aesthetic) que
  puntúan un frame; se **fusionan ponderado** y se comparan contra el umbral de la clase.
- **Consecuencias:** Añadir/quitar señales no toca la orquestación. Refina [D-007].

## D-017 — Preferir APIs sobre librerías pesadas
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** CLIP/aesthetic y whisper local arrastran torch (~2GB).
- **Decisión:** **Gastar en APIs** antes que instalar libs pesadas. El gate usa **Claude visión**;
  CLIP/aesthetic quedan **dormidas** tras el extra opcional `[vision]`. Auto-captions vía **API**
  de transcripción, no whisper local.
- **Consecuencias:** Install ligero por defecto; costo por llamada en vez de cómputo local.
  (Preferencia del usuario.)

## D-018 — Gate suave por defecto (toggle `enforce`)
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** Un gate estricto dispara regeneraciones caras mientras se itera el estilo.
- **Decisión:** `routing.enforce: false` por defecto: el gate **puntúa y registra** pero **no
  bloquea**. `enforce: true` activa reintento/escalado/cola humana.
- **Consecuencias:** Iterar barato; los scores suaves alimentan la calibración futura de umbrales.

## D-019 — Consistencia de personaje API-first (no insightface)
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** El plan original (insightface/ArcFace + clasificador entrenado) choca con [D-017].
- **Decisión:** La consistencia se **logra** propagando referencias del personaje al keyframe
  (`fal-ai/nano-banana/edit`) y se **mide** con Claude visión (`IdentitySignal`). El clasificador
  entrenado va al **backlog**.
- **Consecuencias:** Cero libs pesadas; identidad entra al cache key del keyframe. Validado:
  `char_consistency` real medido por Claude.

## D-020 — Model IDs validados contra fal (no especulativos)
**Estado:** Vigente · **Fecha:** 2026-06-02

- **Contexto:** Los IDs iniciales venían de un documento (cifras ficticias de Jun 2026).
- **Decisión:** Validar cada model ID contra fal antes de usarlo en producción.
- **Consecuencias / Cambios:** `kling-video/v3` → **`v2.1/standard/image-to-video`**;

  Sonda con submit+cancel y payloads distintos para evitar dedupe que dispare costo.
